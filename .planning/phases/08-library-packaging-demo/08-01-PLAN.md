Transform the project from an application build into an npm-publishable library package. The Emscripten build embeds WASM via SINGLE_FILE, a barrel export exposes the public API, Vite library mode bundles everything as ESM, and package.json is configured for npm distribution.

<task type="auto">
  <name>Task 1: Configure SINGLE_FILE build and create library entry point</name>
  <files>
    engine/CMakeLists.txt
    src/index.ts
    tsconfig.lib.json
  </files>
  <action>
1. In `engine/CMakeLists.txt`, add `"-sSINGLE_FILE=1"` to the `target_link_options` block. This embeds the WASM binary as base64 inside engine.js, eliminating the separate .wasm file. After adding, rebuild WASM: `cd engine && emcmake cmake -B build-web && cmake --build build-web -j4`. Verify: the output `engine/build-web/engine.wasm` should NOT exist (or be empty), and `engine/build-web/engine.js` should be ~500-700KB (contains base64 WASM). If the JS file is > 1MB, the SINGLE_FILE + EXPORT_ES6 double-inclusion bug (Emscripten #14309) has triggered — in that case, remove SINGLE_FILE and instead plan to copy the .wasm file to dist. But based on research, Emscripten 4.0.16 likely has this fixed.

2. Create `src/index.ts` as the library barrel export:
```typescript
// Components
export { GlassProvider } from './components/GlassProvider';
export { GlassPanel } from './components/GlassPanel';
export { GlassButton } from './components/GlassButton';
export { GlassCard } from './components/GlassCard';

// Types
export type {
  GlassStyleProps,
  GlassPanelProps,
  GlassButtonProps,
  GlassCardProps,
  GlassColor,
  AccessibilityPreferences,
} from './components/types';

// Hooks (advanced usage)
export { useGlassEngine } from './hooks/useGlassEngine';

// Context types (advanced usage)
export type { GlassContextValue, GlassRegionHandle } from './context/GlassContext';
```

3. Create `tsconfig.lib.json` extending the base tsconfig but enabling declaration emit:
```json
{
  "extends": "./tsconfig.json",
  "compilerOptions": {
    "noEmit": false,
    "declaration": true,
    "declarationMap": true,
    "outDir": "dist"
  },
  "include": ["src"],
  "exclude": ["src/main.tsx", "src/App.tsx"]
}
```

This tsconfig is used by vite-plugin-dts for type generation. It excludes the app entry points (main.tsx, App.tsx) which are not part of the library API.
  </action>

- `ls engine/build-web/engine.wasm` either does not exist or is 0 bytes
- `cat src/index.ts` shows all component and type exports
- `cat tsconfig.lib.json` shows declaration: true, excludes main.tsx and App.tsx
  </verify>
  <done>
WASM is embedded in engine.js via SINGLE_FILE (no separate .wasm file), src/index.ts exports the full public API, and tsconfig.lib.json is ready for declaration generation.

<task type="auto">
  <name>Task 2: Configure Vite library mode and package.json for npm distribution</name>
  <files>
    vite.config.ts
    package.json
  </files>
  <action>
1. Install vite-plugin-dts: `npm install -D vite-plugin-dts`

2. Update `vite.config.ts` to support BOTH library build mode AND the existing dev server. The config should detect build mode via an environment variable or Vite's mode. Use Vite's conditional config:

```typescript
import { defineConfig } from 'vite';
import react from '@vitejs/plugin-react';
import wasm from 'vite-plugin-wasm';
import topLevelAwait from 'vite-plugin-top-level-await';
import liveReload from 'vite-plugin-live-reload';
import dts from 'vite-plugin-dts';
import { resolve } from 'path';

export default defineConfig(({ command, mode }) => {
  // Library build: vite build (default)
  if (command === 'build' && mode !== 'demo') {
    return {
      plugins: [
        react(),
        wasm(),
        topLevelAwait(),
        dts({ tsconfigPath: './tsconfig.lib.json' }),
      ],
      build: {
        lib: {
          entry: resolve(__dirname, 'src/index.ts'),
          formats: ['es'],
          fileName: 'index',
        },
        rollupOptions: {
          external: ['react', 'react-dom', 'react/jsx-runtime'],
          output: {
            globals: {
              react: 'React',
              'react-dom': 'ReactDOM',
              'react/jsx-runtime': 'react/jsx-runtime',
            },
          },
        },
        outDir: 'dist',
        emptyOutDir: true,
      },
    };
  }

  // Dev server or demo build
  return {
    plugins: [
      react(),
      wasm(),
      topLevelAwait(),
      liveReload(['engine/build-web/**/*.{js,wasm}']),
    ],
    optimizeDeps: {
      exclude: ['engine'],
    },
    server: {
      fs: {
        allow: ['..'],
      },
    },
  };
});
```

The key design: `vite build` (no mode flag) builds the library. `vite dev` runs the normal dev server. The demo build (Plan 08-02) will use `--mode demo` or a separate config.

3. Update `package.json`:
- Move react and react-dom from dependencies to peerDependencies
- Add them to devDependencies (for development)
- Add exports field, main, module, types fields
- Add files field to whitelist only dist/
- Add build:lib script
- Add prepublishOnly script
- Keep existing dev and build:wasm scripts

The updated fields (merge with existing, do NOT replace the whole file):

```json
  "module": "./dist/index.js",
  "types": "./dist/index.d.ts",
  "exports": {

    "watch:wasm": "nodemon --watch engine/src -e cpp,h --exec 'npm run build:wasm'",
    "dev:vite": "vite",
    "dev": "concurrently \"npm run watch:wasm\" \"npm run dev:vite\"",
    "build:lib": "vite build",
    "build": "npm run build:wasm && npm run build:lib",
    "prepublishOnly": "npm run build"
  }
}
```

Remove react and react-dom from the top-level `dependencies` object and add them to both `peerDependencies` and `devDependencies`.

Since SINGLE_FILE mode means engine.js is self-contained, Vite should bundle it inline. If it cannot resolve the import, add the engine build-web directory to Vite's resolve configuration or use a Rollup plugin to handle it.
